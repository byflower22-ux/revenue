实现方案对比

两种让迭代标记不影响原始代码的架构方案

方案 A：高阶组件包裹（推荐）

原理

原始组件代码完全不修改。通过一个 IterationMark 包裹组件，在外层添加标记徽章和交互逻辑。包裹关系通过一个独立的配置文件定义。

// 原始代码完全不变
<FilterCard />
// 演示模式下自动变成
<IterationMark version="2.0" date="04-22"
  type="modified" label="筛选条件优化">
  <FilterCard />
</IterationMark>

优势

  • 原始组件零修改，100% 可交付
  • 标记配置集中管理，一目了然
  • 删除标记文件即完全移除

局限

  • 只能标记组件级别的区域
  • 字段级标记需在组件内部加标记点
方案 B：配置驱动 + 代码注入

原理

在原始组件内部用 MARK() 函数标记位置。纯净模式下该函数为空操作，不输出任何内容；演示模式下渲染标记徽章。

// 原始代码中加入标记点
{MARK('2.0', '04-22', 'new')}
<Input placeholder="新字段" />
// 纯净模式: MARK = () => null
// 演示模式: MARK 渲染徽章

优势

  • 字段级精确标记
  • 标记就在代码旁边，好维护
  • 一行代码切换模式

局限

  • 原始代码中多了 MARK() 调用
  • 交付时需确认已关闭演示模式
  • MARK 调用散布在各文件中

推荐：方案 A（高阶组件包裹）

对于字段级标记的需求，可以用混合策略：区域标记用高阶组件包裹，字段级标记用轻量的 data-mark 属性标注——纯净模式下这些属性无任何效果，前端开发可选择性保留或删除。
